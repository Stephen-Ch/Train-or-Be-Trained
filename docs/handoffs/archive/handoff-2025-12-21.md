### The Problem

### Suspected Root Causes
1. **Service Worker caching** — PWA SW caches aggressively; even hard refresh doesn't bypass
2. **Multiple node processes** — Dev servers left running interfere
3. **Browser cache** — May need full cache clear, not just hard refresh
4. **Incognito fallback** — Not consistently used

### What's Been Tried (All Failed)
- Hard refresh (Ctrl+Shift+R)
- sessionStorage.clear()
- Closing/reopening tabs
- Stopping ng serve, restarting
- Serving prod build on new ports (5005, 5006, 5007, 4200, 4201, 8080)

### Debug Tools Now Available

**Use `?debugQuestion=1` in URL:**
```
http://localhost:4200/q/liberty?debugQuestion=1
```

This shows a yellow overlay with:
- `debugSource`: "query" | "session" | "off" (confirms sticky flag working)
- Current URL
- `currentId`, `phase`, `selectedOption`
- `currentFollowUpIndex`, `totalFollowUpsForSelected`, `answeredFollowUps`
- `canContinue`, `options`

**Console output on Continue click:**
- `CONTINUE_CLICK` — state snapshot
- `CONTINUE_BRANCH` — which code path taken
- `NAVIGATE_TO` / `NAVIGATE_RESULT` — navigation outcomes

**The flag persists in sessionStorage** — once you load with `?debugQuestion=1`, subsequent navigations (even without the param) will show the overlay.

## Answer Key Convention (IMPORTANT)

| Question Type | Key Format | Example |
|--------------|------------|---------|
| TLQ | `{tlqId}` | `liberty-q0` |
| Followup | `fu:{categoryId}:{tlqId}:{index}` | `fu:liberty:liberty-q0:0` |

The `fu:` prefix prevents followup answers from colliding with TLQ answers (since the content model reuses the same `id`).
